## 1. ОБЩЕЕ ОПИСАНИЕ

### 1.1. Назначение системы

Telegram-бот для автоматизированного управления VPN сервером на базе протокола AmneziaWG (WireGuard с обфускацией). Система предоставляет:

- Генерацию и управление VPN ключами для пользователей
- Бесплатную модель доступа: пользователи активируются администратором
- Развитое взаимодействие пользователей с администратором (вопросы, запросы, обратная связь)
- Режим открытого доступа (временное отключение авторизации)
- Административное управление пользователями, ключами и настройками
- Мониторинг трафика в реальном времени
- Веб-панель для расширенного администрирования с конфигурацией бота через UI

### 1.2. Целевая аудитория

| Роль | Описание | Функции |
|------|----------|---------|
| **Пользователь** | Друзья/знакомые администратора, активированные вручную или через режим открытого доступа | Создание ключей в рамках лимита, общение с админом, просмотр статуса |
| **Администратор** | Владелец бота | Полное управление системой, пользователями, ключами, настройками |

### 2.2. Потоки данных

| Поток | Источник | Назначение | Протокол | Описание |
|-------|----------|-----------|----------|----------|
| Управление VPN | Bot → VPN Server | SSH/Docker exec | Создание/удаление ключей через wg CLI |

### 2.3. Модель взаимодействия бота с VPN сервером

VPN Manager автоматически определяет режим работы:

- **Локальный режим** — если бот запущен на том же сервере, что и Docker контейнер AmneziaWG, используется прямой `docker exec` (быстрее, нет overhead SSH)
- **Удаленный режим** — если бот на другом сервере, используется SSH + `docker exec` через Paramiko

Детекция выполняется при инициализации `VPNManager`: проверяется наличие контейнера через `docker ps`, а также сравнение `SERVER_HOST` с локальными адресами.

## 4. БАЗА ДАННЫХ

### 4.1. Общие настройки

- **СУБД:** SQLite 3 с WAL (Write-Ahead Logging) mode
- **ORM:** SQLAlchemy 2.0 с Declarative Base
- **Пул соединений:** QueuePool (pool_size=10, max_overflow=20)
- **Timeout:** 30 секунд для операций, busy_timeout=30000 для блокировок
- **Retry:** Автоматические повторы при `OperationalError` и `DatabaseError` (до 3 попыток)

**Оптимизации SQLite при инициализации:**
```sql
PRAGMA journal_mode=WAL;       -- Параллельное чтение и запись
PRAGMA synchronous=NORMAL;     -- Баланс скорости и надежности
PRAGMA busy_timeout=30000;     -- 30 секунд timeout блокировок
PRAGMA foreign_keys=ON;        -- Включение внешних ключей
PRAGMA temp_store=MEMORY;      -- Временные таблицы в памяти
PRAGMA mmap_size=268435456;    -- 256MB для memory-mapped I/O
```

**Важно:** Значения `bytes_received` и `bytes_sent` являются **накопительными** (так возвращает `wg show dump`). Для вычисления трафика за период необходимо брать разницу между последовательными snapshot'ами.

### 4.8. Миграции

Миграции выполняются автоматически при инициализации БД (`init_db()`). Система проверяет наличие колонок через `PRAGMA table_info()` и добавляет отсутствующие через `ALTER TABLE ... ADD COLUMN`. Это обеспечивает совместимость при обновлении.

## 5. АВТОРИЗАЦИЯ И РЕГИСТРАЦИЯ ПОЛЬЗОВАТЕЛЕЙ

### 5.1. Модель доступа

Все пользователи — бесплатные. Активация выполняется одним из двух способов:

1. **Ручная активация** — администратор активирует пользователя вручную, задаёт лимит `max_keys`
2. **Режим открытого доступа** — при включённом режиме все новые пользователи автоматически активируются при `/start`

### 5.2. Режим открытого доступа (Auth Bypass)

Администратор может временно отключить авторизацию, чтобы при перезапуске бота не получать десятки запросов на активацию.

**Хранение:** Таблица `app_config`
- `auth_bypass_enabled` — `"true"` / `"false"`
- `auth_bypass_until` — ISO datetime (дата и время окончания)
- `auth_bypass_max_keys` — количество ключей для авто-активированных пользователей (по умолчанию `"1"`)

**Поведение при включённом режиме:**
1. Любой пользователь (новый или существующий неактивный), отправивший `/start`, автоматически активируется (`is_active=True`, `max_keys=auth_bypass_max_keys`)
2. Запрос на активацию администратору **не отправляется**
3. Бот проверяет `auth_bypass_until` — если время истекло, режим автоматически выключается
4. Во всех интерфейсах администратора отображается индикатор: «🔓 Открытый доступ до ДД.ММ ЧЧ:ММ»
5. Администратор получает уведомление, когда режим истекает

**Управление через Telegram:**
```
🔓 Режим открытого доступа
├── Текущий статус: ✅ Включён до 15.06 18:00 / ❌ Выключен
├── Включить на 12 часов
├── Включить на 1 день
├── Включить на 2 дня
├── Включить на 3 дня
└── ❌ Выключить сейчас
```

**Управление через веб-панель:** На странице Settings — переключатель с выбором длительности и обратный отсчёт.

### 5.3. Процесс регистрации нового пользователя

```
Пользователь отправляет /start
         │
         ▼
   Поиск в БД по telegram_id
         │
    ┌─────┴─────┐
    │ Найден?   │
    └─────┬─────┘
     Да   │   Нет
     │    │    │
     │    │    ▼
     │    │ Создание нового пользователя
     │    │    │
     │    │    ▼
     │    │ Проверка: режим открытого доступа?
     │    │    │
     │    │ ┌──┴──┐
     │    │ │ Да  │ → Авто-активация (is_active=True, max_keys=N)
     │    │ │ Нет │ → Неактивный (is_active=False, max_keys=0)
     │    │ └─────┘
     │    │
     ▼    │
  Для существующих неактивных пользователей:
  Проверка: режим открытого доступа?
     │
  ┌──┴──┐
  │ Да  │ → Авто-активация (is_active=True, max_keys=N)
  │ Нет │ → Без изменений
  └─────┘
     │    │
     ▼    │

     │

  ┌──┴──┐

  └─────┘
```

**Меню неактивного пользователя:**
- 🔓 Запросить активацию
- 📱 Предоставить телефон
- ❓ Задать вопрос

### 5.4. Авторизация по номеру телефона

При отправке контакта через Telegram:

1. Номер нормализуется: `89001234567` → `+79001234567`
2. Выполняется поиск в БД по нормализованному номеру
3. При нахождении дублей (по telegram_id и phone) — объединение записей
4. Обновление данных пользователя (telegram_id, phone, username и т.д.)
5. Проверка `is_active` для определения уровня доступа

**Нормализация телефона:**
- `+79001234567` → `+79001234567` (без изменений)
- `89001234567` → `+79001234567` (замена 8 на +7)
- `79001234567` → `+79001234567` (добавление +)
- `+89001234567` → `+79001234567` (замена +8 на +7)

### 5.5. Автоверификация при запросе активации

При нажатии «🔓 Запросить активацию» выполняется автоматическая проверка:

1. **Поиск по telegram_id** — среди активных пользователей с `is_active=True`
2. **Поиск по phone_number** — по нормализованному номеру
3. **Поиск по username** — case-insensitive сравнение
4. **Поиск среди пользователей без telegram_id** — для случаев когда пользователь был добавлен по телефону через админ-панель

Если совпадение найдено → автоматическая активация с копированием `max_keys`.
Если совпадение не найдено → отправка запроса администратору с кнопками:
- Выбор количества ключей (1, 3, 5, ±1)
- ✅ Активировать / ❌ Отказать

### 5.6. Определение администратора

- Администратор определяется по `ADMIN_ID` из `.env` файла или `app_config` в БД
- Флаг `is_admin` в БД синхронизируется автоматически при каждом `/start`
- Проверка `user.id == ADMIN_ID` выполняется при каждом действии в админ-панели

## 6. УПРАВЛЕНИЕ VPN КЛЮЧАМИ

### 6.1. Жизненный цикл ключа

```
  Запрос создания ключа
         │
         ▼
  Проверка лимитов:
  • active_keys < max_keys?
         │
         ▼
  Генерация WireGuard ключей:
  • Приватный ключ: wg genkey
  • Публичный ключ: wg pubkey
  (локально или через Docker/SSH на сервере)
         │
         ▼
  Получение следующего свободного IP:
  • wg show wg0 dump → парсинг used IPs
  • Итерация по 10.8.1.0/24 (начиная с .2)
         │
         ▼
  Получение публичного ключа сервера:
  • wg show wg0 public-key
         │
         ▼
  Добавление peer на сервер:
  • wg set wg0 peer <pubkey> allowed-ips <IP>/32
  • Верификация: wg show wg0 dump → проверка наличия peer
         │
         ▼
  Генерация конфигурации (.conf)
         │
         ▼
  Генерация QR-кода (.png)
         │
         ▼
  Сохранение в БД (VPNKey)
         │
         ▼
  Отправка пользователю:
  • Файл .conf
  • QR-код (.png)
  • Текст конфигурации (для копирования)
  • Инструкция по установке
```

### 6.2. Формат конфигурации AmneziaWG

```ini
[Interface]
PrivateKey = <приватный_ключ_клиента>
Address = 10.8.1.X/32
DNS = 1.1.1.1, 8.8.8.8

[Peer]
PublicKey = <публичный_ключ_сервера>
Endpoint = <SERVER_HOST>:40680
AllowedIPs = 0.0.0.0/0
PersistentKeepalive = 25
```

### 6.3. Формат имени ключа

```
{user_name}_{phone_part}_{date}_{telegram_id}
```

Пример: `Aleksei_plus79001_20250601_1230_2018301997`

- `user_name` — имя пользователя (до 15 символов, пробелы → `_`)
- `phone_part` — номер телефона (до 10 символов, `+` → `plus`)
- `date` — дата и время создания (формат: `YYYYMMDD_HHMM`)
- `telegram_id` — Telegram ID пользователя
- Общая длина ограничена 60 символами

### 6.4. Удаление ключа

1. Удаление peer с сервера: `wg set wg0 peer <pubkey> remove`
2. Сохранение конфигурации WireGuard: `wg-quick save wg0`
3. Удаление файла .conf из `vpn_configs/`
4. Удаление QR-кода (.png) из `vpn_configs/`
5. Удаление записи из БД

### 6.5. Синхронизация ключей с сервером

Функция `sync_keys_with_server()` сравнивает ключи в БД с пирами на сервере:

- **Ключи на сервере, но не в БД** → создание записи в БД с `created_by_bot=False`, привязка к системному пользователю (telegram_id=0)
- **Ключи в БД, но не на сервере** → пометка `is_active=False`

### 6.6. Асинхронное выполнение

Все блокирующие операции (SSH, Docker exec) выполняются в `ThreadPoolExecutor` (max_workers=5) через `asyncio.run_in_executor()`, чтобы не блокировать event loop Telegram бота.

## 7. ВЗАИМОДЕЙСТВИЕ ПОЛЬЗОВАТЕЛЕЙ С АДМИНИСТРАТОРОМ

### 7.1. Обзор

Система предоставляет пользователям набор инструментов для общения с администратором прямо через бота. Все обращения сохраняются в таблице `user_messages` и доступны администратору через бота и веб-панель.

### 7.2. Меню активного пользователя

```
📱 Главное меню
├── 🔐 Получить AmneziaWG ключ    — Создание нового VPN ключа
├── 📋 Мои ключи                   — Список ключей с возможностью удаления
├── 📊 Статус подключения           — Информация о ключах и трафике
├── 🔑 Запросить ещё ключ           — Запрос увеличения лимита ключей
├── ❓ Задать вопрос                — Написать сообщение администратору
├── 🐛 Сообщить о проблеме          — Сообщить о технической проблеме
├── ℹ️ Помощь                       — Справка и инструкции
└── ⚙️ Админ-панель                 — (только для администратора)
```

### 7.3. Функция «📊 Статус подключения»

Самостоятельная проверка без участия администратора. Показывает:

- Количество активных ключей / лимит (`2 из 3`)
- Для каждого ключа:
  - Имя ключа
  - IP адрес
  - Статус: 🟢 Подключён / 🔴 Не активен (на основе last_handshake < 5 мин)
  - Последний handshake (время)
  - Трафик за сегодня (↑ отправлено / ↓ получено)
- Общий трафик за месяц

### 7.4. Функция «🔑 Запросить ещё ключ»

Доступна, когда пользователь достиг лимита `max_keys`. Процесс:

1. Пользователь нажимает «🔑 Запросить ещё ключ»
2. Если `active_keys < max_keys` — бот предлагает сначала создать ключ через «🔐 Получить ключ»
3. Если `active_keys >= max_keys` — бот запрашивает причину (текстовое сообщение)
4. Запрос сохраняется в `user_messages` (`message_type='key_request'`)
5. Администратор получает уведомление с информацией:
   - Имя пользователя, Telegram ID
   - Текущее кол-во ключей / лимит
   - Причина запроса
   - Кнопки: «➕ Увеличить на 1», «➕ Увеличить на 3», «❌ Отклонить»
6. Пользователь получает уведомление о результате

### 7.5. Функция «❓ Задать вопрос»

Прямой канал связи с администратором:

1. Пользователь нажимает «❓ Задать вопрос»
2. Бот просит ввести текст сообщения (до 1000 символов)
3. Сообщение сохраняется в `user_messages` (`message_type='question'`)
4. Администратор получает сообщение с:
   - Информацией о пользователе
   - Текстом вопроса
   - Кнопкой «💬 Ответить»
5. При нажатии «💬 Ответить» бот переходит в режим ожидания ответа от администратора
6. Ответ сохраняется в `admin_reply`, пользователь получает уведомление

**Доступно также для неактивных пользователей** — они могут задать вопрос до активации.

### 7.6. Функция «🐛 Сообщить о проблеме»

Структурированная форма для отправки технических проблем:

1. Пользователь нажимает «🐛 Сообщить о проблеме»
2. Выбор категории проблемы:
   - 🔌 VPN не подключается
   - 🐌 Низкая скорость
   - ⚡ Обрывы соединения
   - 📱 Проблема с приложением
   - 🔧 Другое
3. Бот просит добавить описание (опционально)
4. Запрос сохраняется в `user_messages` (`message_type='problem'`)
5. Администратор получает уведомление с категорией и описанием
6. Кнопки: «💬 Ответить», «✅ Решено»

### 7.7. Функция «📢 Рассылка» (только админ)

Администратор может отправить сообщение всем пользователям:

1. В админ-панели выбрать «📢 Рассылка»
2. Выбор аудитории:
   - 👥 Все активные пользователи
   - 👥 Все пользователи (включая неактивных)
3. Ввод текста рассылки (поддержка Markdown)
4. Предпросмотр сообщения
5. Подтверждение отправки
6. Прогресс: «Отправлено: 15/42»

### 7.8. Входящие сообщения администратора

В админ-панели бота раздел «📨 Входящие»:

```
📨 Входящие сообщения (3 новых)
├── ❓ Вопрос от Алексей: «Как настроить на iOS?» — 10 мин назад
├── 🔑 Запрос ключа от Мария: «Нужен для ноутбука» — 1 час назад
└── 🐛 Проблема от Иван: «VPN не подключается» — 2 часа назад
```

При нажатии на сообщение — развёрнутый вид с кнопками действий.

3. Формирует URL: `http://domain:8889/?token=<token>`

## 10. МОНИТОРИНГ ТРАФИКА

### 10.1. Сбор данных

**Источник:** Команда `wg show wg0 dump` в Docker контейнере AmneziaWG.

**Формат вывода wg dump** (поля разделены табуляцией):
```
public_key  preshared_key  endpoint  allowed_ips  last_handshake  transfer_rx  transfer_tx  persistent_keepalive
```

**Расписание:** Каждые 15 минут через systemd timer (`vpn-traffic-snapshots.timer`).

**Процесс:**
1. Скрипт `sync_traffic_snapshots.py` запускается по таймеру
2. Получает данные из WireGuard через `traffic_manager.get_wireguard_stats()`
3. Для каждого peer'а с совпадающим public_key в БД создает snapshot в `traffic_statistics`
4. Значения `bytes_received/bytes_sent` — накопительные (как выдает WireGuard)

### 10.2. Вычисление трафика за период

Так как WireGuard возвращает накопительные значения, для вычисления трафика за интервал используется разница между последовательными snapshot'ами:

```
Трафик за интервал = Snapshot[N] − Snapshot[N−1]
```

Если snapshot'а до начала периода нет — первый snapshot в периоде используется как базовый (его трафик не учитывается, чтобы не показывать весь накопленный объём).

### 10.3. Графики (периоды и интервалы)

| Период | Интервал группировки | Формат метки |
|--------|---------------------|--------------|
| 6 часов | 15 минут | HH:MM |
| Сутки | 1 час | HH:00 |
| Неделя | 3 часа | DD.MM HH:00 |
| Месяц | 1 день | DD.MM |

### 10.4. Определение активного подключения

Peer считается активным, если `last_handshake` был менее 5 минут назад:
```python
is_active = (now - last_handshake).total_seconds() < 300
```

## 12. БЕЗОПАСНОСТЬ

### 12.1. Авторизация

| Компонент | Метод | Описание |
|-----------|-------|----------|
| Telegram бот | telegram_id | Каждый пользователь идентифицируется по Telegram ID |
| Админ-панель бота | ADMIN_ID | Проверка при каждом действии |
| Веб-панель | Токен (24ч) | Генерируется через бота, передается в URL |
| Режим открытого доступа | app_config | Временно обходит авторизацию для всех пользователей |

### 12.2. Защита данных

- Приватные ключи WireGuard хранятся в БД (рекомендуется шифрование в продакшене)
- Секретные настройки (`is_secret=True`) маскируются в веб-интерфейсе
- BOT_TOKEN маскируется при отображении (показываются только первые/последние символы)
- SSH ключи для доступа к серверу хранятся локально
- `.env` файл не должен попадать в git (добавлен в .gitignore)

### 12.3. Ограничения доступа

- Лимит ключей на пользователя (настраивается администратором)
- Неактивные пользователи не могут создавать ключи (кроме режима открытого доступа)
- Все callback-запросы проверяют авторизацию перед выполнением
- Устаревшие callback-запросы (>48ч) обрабатываются gracefully с подсказкой пользователю
- Режим открытого доступа автоматически деактивируется по истечении времени

### 12.4. Обработка устаревших запросов

Телеграм может отправлять callback-запросы от старых сообщений. Система:
- Перехватывает ошибки `query is too old`, `response timeout expired`
- Отправляет пользователю подсказку: «⚠️ Эта кнопка устарела. Откройте меню через '📱 Меню'»
- Не прерывает работу бота из-за устаревших запросов

## 13. ОБРАБОТКА ОШИБОК

### 13.1. Глобальный обработчик ошибок

Класс `VPNBot` содержит `_application_error_handler`, который:
- Логирует все необработанные ошибки
- Игнорирует устаревшие callback-запросы (`BadRequest`)
- Предупреждает о конфликтах polling (`Conflict` — два экземпляра бота)
- Добавляет паузу при сетевых ошибках (`TimedOut`, `NetworkError`)

### 13.2. Retry-логика

- **HTTP запросы:** `tenacity` — до 3 попыток с экспоненциальной задержкой (0.5–5 сек)
- **Сессии БД:** `get_db_session()` — до 3 попыток при `OperationalError`/`DatabaseError`
- **Callback ответы:** `_safe_answer_callback()` — обработка устаревших запросов
- **Редактирование сообщений:** `_safe_edit_message_text()` — обработка «message is not modified» и устаревших запросов

### 13.3. Ошибки создания ключа

| Ошибка | Сообщение пользователю |
|--------|----------------------|
| SSH / подключение | «Не удалось подключиться к серверу» |
| Публичный ключ сервера | «Не удалось получить публичный ключ, проверьте Docker» |
| IP адреса | «Нет доступных IP адресов» |
| Другое | «Ошибка при генерации ключа, детали в логе» |
