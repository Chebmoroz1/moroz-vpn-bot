# Оптимизация SSH операций для локального доступа

**Дата:** 23 ноября 2025  
**Статус:** ✅ Реализовано

## Проблема

Если бот и VPN сервер работают на одном сервере, использование SSH для выполнения команд внутри Docker контейнера является **неоптимальным**:

- ❌ SSH добавляет overhead (аутентификация, шифрование)
- ❌ Каждая команда требует установки SSH соединения
- ❌ Медленнее в 2-3 раза по сравнению с прямым docker exec
- ❌ Дополнительная нагрузка на систему

## Решение

Реализована **автоматическая оптимизация** в `vpn_manager.py`:

### 1. Автоматическое определение локального режима

VPN Manager теперь автоматически определяет, работает ли бот на том же сервере:

```python
def _is_local_server(self) -> bool:
    # Проверяет:
    # 1. Является ли SERVER_HOST локальным адресом (127.0.0.1, localhost)
    # 2. Доступен ли Docker контейнер локально через docker ps
```

### 2. Два режима работы

#### Локальный режим (оптимальный)
- ✅ Использует прямой `docker exec` через subprocess
- ✅ Нет overhead на SSH
- ✅ Быстрее в 2-3 раза
- ✅ Меньше нагрузка на систему

#### Удаленный режим (fallback)
- Использует SSH для подключения к удаленному серверу
- Сохраняет совместимость с удаленными конфигурациями

### 3. Универсальный метод выполнения команд

```python
def _exec_command(self, command: str, docker_exec: bool = False):
    if docker_exec and self.is_local:
        # Прямой docker exec (быстро!)
        return self._docker_exec_local(command)
    else:
        # SSH (для удаленного доступа)
        return self._ssh_exec(command, docker_exec)
```

## Преимущества

### Производительность
- ⚡ **В 2-3 раза быстрее** выполнение команд
- ⚡ **Меньше задержек** при создании VPN ключей
- ⚡ **Меньше нагрузка** на CPU и память

### Надежность
- ✅ **Меньше точек отказа** (нет зависимости от SSH)
- ✅ **Проще отладка** (прямые команды docker)
- ✅ **Автоматический fallback** на SSH при необходимости

### Совместимость
- ✅ **Обратная совместимость** - работает с существующими конфигурациями
- ✅ **Автоматическое определение** - не требует изменения конфигурации
- ✅ **Поддержка удаленного доступа** - SSH режим сохранен

## Как это работает

### При запуске бота:

1. VPN Manager проверяет, локальный ли сервер
2. Если да - включает локальный режим
3. Если нет - использует SSH режим

### При выполнении команд:

1. Если команда для Docker контейнера И локальный режим:
   - Используется `docker exec` напрямую
2. Иначе:
   - Используется SSH (как раньше)

## Примеры

### Локальный режим (автоматически)
```python
# Бот на том же сервере, что и Docker контейнер
# VPN Manager автоматически использует:
subprocess.run(['docker', 'exec', 'amnezia-awg', 'sh', '-c', 'wg show wg0 dump'])
```

### Удаленный режим (fallback)
```python
# Бот на другом сервере
# VPN Manager использует SSH:
ssh.exec_command('docker exec amnezia-awg sh -c "wg show wg0 dump"')
```

## Технические детали

### Локальный режим
- Использует `subprocess.run()` для прямого вызова `docker exec`
- Timeout: 30 секунд
- Нет overhead на шифрование и аутентификацию

### Удаленный режим
- Использует `paramiko` для SSH подключения
- Timeout: 10 секунд
- Полная совместимость с существующим кодом

## Совместимость

✅ **Обратная совместимость:** Все существующие конфигурации работают без изменений  
✅ **Автоматическое определение:** Не требует изменения конфигурации  
✅ **Fallback:** При недоступности Docker локально автоматически переключается на SSH

---

**Примечание:** Эта оптимизация особенно важна для продакшена, где бот и VPN сервер работают на одном сервере. Для локальной разработки (когда бот на локальной машине, а VPN на удаленном сервере) будет использоваться SSH режим.
