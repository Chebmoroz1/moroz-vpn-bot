# Отчет об исправлении выбросов в графике трафика

## Проблема

В графике трафика наблюдался большой выброс в столбце **14:00** (45757.58 MB), а также выбросы в **15:00** (1335.47 MB) и **16:00** (699.67 MB).

## Причина

Выбросы связаны с тем, что snapshots начали собираться только **23 ноября 2025 в 14:06**. Первый snapshot для каждого ключа содержит весь накопленный трафик с начала работы ключа, а не трафик за конкретный интервал.

### Детали

1. **Первый snapshot**: 2025-11-23 14:06:26
2. **Количество ключей**: 31
3. **Все ключи получили первый snapshot в 14:06** (в интервале 14:00)
4. **Накопленный трафик в первом snapshot**: от 0 до 7.60 GB для разных ключей

### Проблема в логике расчета

При вычислении разницы между snapshots:
- Для интервала 14:00: первый snapshot содержал весь накопленный трафик (например, 7.60 GB)
- Разница вычислялась от 0 (так как не было предыдущего snapshot'а)
- Результат: весь накопленный трафик приписывался интервалу 14:00

## Решение

Исправлена логика в методе `_get_chart_data_from_snapshots()`:

1. **Определение первого snapshot'а**: Для каждого ключа определяется первый snapshot в периоде
2. **Пропуск первого snapshot'а**: Первый snapshot пропускается при расчете трафика (не добавляется к интервалу)
3. **Обновление базовых значений**: Первый snapshot используется как базовое значение для последующих расчетов

### Изменения в коде

```python
# Определяем первый snapshot для каждого ключа
first_snapshot_timestamps = {}

for key_id in unique_key_ids:
    # Ищем последний snapshot ДО начала периода
    first_snapshot_before = db.query(TrafficStatistics).filter(...)
    
    if first_snapshot_before:
        # Используем значение до начала периода как базовое
        initial_key_data[key_id] = {...}
    else:
        # Если нет snapshot'а до начала периода, находим первый в периоде
        first_snapshot_in_period = db.query(TrafficStatistics).filter(...)
        if first_snapshot_in_period:
            # Сохраняем как базовое значение
            initial_key_data[key_id] = {...}
            # Запоминаем timestamp для пропуска
            first_snapshot_timestamps[key_id] = first_snapshot_in_period.timestamp

# При расчете трафика для интервала
for key_id, snapshot_data in interval_snapshots.items():
    is_first_snapshot = (
        key_id in first_snapshot_timestamps and
        abs((snapshot_data['timestamp'] - first_snapshot_timestamps[key_id]).total_seconds()) < 60
    )
    
    if is_first_snapshot:
        # Пропускаем первый snapshot - просто обновляем базовые значения
        prev_key_data[key_id] = {...}
    else:
        # Вычисляем разницу от предыдущего значения
        key_received = max(0, snapshot_data['received'] - prev_key_data[key_id]['received'])
        ...
```

## Результаты

### До исправления:
- 14:00: **45757.58 MB** (выброс)
- 15:00: **1335.47 MB** (выброс)
- 16:00: **314.97 MB** (выброс)

### После исправления:
- 14:00: **0.00 MB** (выброс устранен)
- 15:00: **1335.47 MB** (остается, требует дополнительной проверки)
- 16:00: **699.67 MB** (остается, требует дополнительной проверки)

## Статус

✅ **Выброс в 14:00 устранен**

⚠️ **Выбросы в 15:00 и 16:00 остаются** - требуют дополнительного анализа

### Примечание

Выбросы в 15:00 и 16:00 могут быть связаны с:
1. Новыми ключами, которые начали работать в эти интервалы
2. Проблемами в логике обновления `prev_key_data` между интервалами
3. Неправильным определением первого snapshot'а для некоторых ключей

Требуется дополнительная проверка логики для этих интервалов.
