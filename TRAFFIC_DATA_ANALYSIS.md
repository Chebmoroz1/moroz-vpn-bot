# Анализ данных трафика и план исправления

## Проблема

### Текущая ситуация

1. **WireGuard хранит накопительные значения** - общий трафик с момента создания ключа
   - Total WG: 47,455,765,716 bytes (45,257.34 MB) - это НАКОПИТЕЛЬНЫЙ трафик

2. **В БД сохраняются накопительные значения** (не дневные!)
   - БД за сегодня: 47,423,287,564 bytes (45,226.37 MB)
   - Это те же накопительные значения из WireGuard

3. **График показывает накопительные значения, распределенные искусственно**
   - Для "6hours": 45GB распределены по 24 интервалам (15 минут) с помощью синусоиды
   - Для "day": те же 45GB распределены по 24 часам с помощью синусоиды
   - **Это НЕ реальный трафик за период, а накопительный трафик, распределенный искусственно!**

### Таблица сравнения данных

| Источник | Значение | Тип данных | Проблема |
|----------|----------|------------|----------|
| WireGuard (текущее) | 45,257 MB | Накопительный (с момента создания ключа) | ✅ Корректно |
| БД за сегодня | 45,226 MB | Накопительный (перезаписывается при sync) | ❌ Не дневной трафик |
| График "6hours" | 45,226 MB | Накопительный, распределенный по синусоиде | ❌ Не реальный трафик за 6 часов |
| График "day" | 45,226 MB | Накопительный, распределенный по синусоиде | ❌ Не реальный трафик за сутки |

### Пример проблемы

**График "day" показывает:**
- 00:00: 831.63 MB
- 01:00: 831.63 MB
- 02:00: 831.63 MB
- ...
- 23:00: 831.63 MB

Это означает, что график показывает одинаковые значения для всех часов, что не соответствует реальности.

## Корневая причина

1. **`sync_traffic_stats()` перезаписывает значения в БД** (строки 214-215 в `traffic_manager.py`)
   - При каждом вызове накопительные значения из WireGuard просто перезаписываются
   - Нет истории изменений
   - Нет возможности вычислить трафик за период

2. **В БД хранится только дата, без времени**
   - Невозможно отследить изменения в течение дня
   - Невозможно вычислить трафик за короткие периоды (6 часов, сутки)

3. **График использует искусственное распределение**
   - Синусоидальная функция распределяет накопительный трафик по интервалам
   - Это не отражает реальную нагрузку на сервер

## План исправления

### Вариант 1: Хранение snapshots трафика (РЕКОМЕНДУЕТСЯ)

**Преимущества:**
- Точные данные по интервалам
- Возможность вычисления реального трафика за любой период
- История изменений

**Реализация:**

1. **Создать новую таблицу `traffic_snapshots`:**
   ```sql
   CREATE TABLE traffic_snapshots (
       id INTEGER PRIMARY KEY,
       vpn_key_id INTEGER NOT NULL,
       timestamp DATETIME NOT NULL,
       bytes_received BIGINT DEFAULT 0,
       bytes_sent BIGINT DEFAULT 0,
       last_handshake DATETIME,
       FOREIGN KEY (vpn_key_id) REFERENCES vpn_keys(id)
   );
   CREATE INDEX idx_traffic_snapshots_key_timestamp ON traffic_snapshots(vpn_key_id, timestamp);
   ```

2. **Модифицировать `sync_traffic_stats()`:**
   - При каждой синхронизации сохранять snapshot с текущим временем
   - Сохранять накопительные значения из WireGuard
   - Ограничить количество snapshots (хранить последние 7 дней, удалять старые)

3. **Модифицировать `get_chart_data()`:**
   - Для коротких периодов (6 часов, сутки): использовать snapshots
   - Вычислять разницу между snapshots для получения трафика за интервал
   - Для длительных периодов (неделя, месяц): использовать данные из БД (накопительные) и вычислять разницу между днями

4. **Добавить периодическую синхронизацию:**
   - Запускать `sync_traffic_stats()` каждые 15 минут (через cron или systemd timer)
   - Это обеспечит snapshots для коротких периодов

### Вариант 2: Вычисление разницы между днями (БЫСТРОЕ РЕШЕНИЕ)

**Преимущества:**
- Минимальные изменения в коде
- Работает для длительных периодов (неделя, месяц)

**Недостатки:**
- Не работает для коротких периодов (6 часов, сутки)
- Нет точных данных по интервалам

**Реализация:**

1. **Модифицировать `get_chart_data()`:**
   - Для периода "month": вычислять разницу между днями
   - Для периода "week": вычислять разницу между днями и распределять по 3-часовым интервалам
   - Для периодов "6hours" и "day": использовать данные из WireGuard и вычислять разницу от предыдущего дня

2. **Проблема:** Для "6hours" и "day" все равно нужны snapshots или данные из WireGuard с вычислением разницы

### Вариант 3: Гибридный подход (РЕКОМЕНДУЕТСЯ ДЛЯ БЫСТРОГО ВНЕДРЕНИЯ)

**Реализация:**

1. **Для коротких периодов (6 часов, сутки):**
   - Использовать данные из WireGuard напрямую
   - Вычислять разницу между текущим значением и значением в начале периода
   - Для этого нужно сохранять snapshot в начале дня или использовать данные из БД за вчера

2. **Для длительных периодов (неделя, месяц):**
   - Использовать данные из БД
   - Вычислять разницу между днями
   - Распределять по интервалам с учетом времени суток (но без синусоиды, а на основе реальных данных)

3. **Модифицировать `sync_traffic_stats()`:**
   - Сохранять snapshot в начале дня (00:00) или при первом вызове
   - Для вычисления дневного трафика: разница между текущим значением и значением в начале дня

## Рекомендуемый план действий

### Этап 1: Быстрое исправление (1-2 часа)

1. **Исправить `get_chart_data()` для периода "month":**
   - Вычислять разницу между днями
   - Показывать реальный трафик за каждый день

2. **Исправить `get_chart_data()` для периода "week":**
   - Вычислять разницу между днями
   - Распределять дневной трафик по 3-часовым интервалам (равномерно или на основе реальных данных, если есть)

3. **Для периодов "6hours" и "day":**
   - Использовать данные из WireGuard
   - Вычислять разницу от значения в начале дня (из БД за сегодня, если есть, или использовать данные за вчера как базовую точку)

### Этап 2: Долгосрочное решение (4-6 часов)

1. **Создать таблицу `traffic_snapshots`**
2. **Модифицировать `sync_traffic_stats()` для сохранения snapshots**
3. **Добавить периодическую синхронизацию (каждые 15 минут)**
4. **Модифицировать `get_chart_data()` для использования snapshots**
5. **Добавить очистку старых snapshots (старше 7 дней)**
